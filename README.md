# AI 开发者协作指南 (AI Collaboration & Workflow Rules)

你好，AI 助手。作为参与本 Godot 游戏项目的核心协作者，在执行任何指令前，你必须严格阅读并遵守以下开发规范与工作流机制：

## 一、 代码质量与开发声明 (Code Quality Declaration)
你所输出的所有代码和架构设计，必须符合工业级标准及 Godot 引擎的最佳实践：
1. **规范性**：严格遵循 C# 代码规范（类/方法/属性使用 `PascalCase`，局部变量/参数使用 `camelCase`，私有字段使用 `_camelCase`），遵循 Godot C# 绑定的命名约定。
2. **解耦与模块化**：严禁写“面条代码”。必须合理利用 Godot 的信号 (Signals)、分组 (Groups) 和状态机，保持场景与脚本的高内聚、低耦合。
3. **健壮性**：所有节点获取、类型转换和外部输入都必须进行 null 检查和有效性验证，避免抛出异常。
4. **可读性**：复杂逻辑、数学计算和特定业务机制必须配有清晰的中文注释，说明“为什么这么做”而非仅仅是“做了什么”。

## 二、 知识库读取与强制初始化 (Knowledge Base & Initialization)
在开始任何具体的开发任务或回答问题前，你必须进行以下环境上下文同步：
1. **读取核心规则**：必须优先读取根目录下的 `Game_Rules` 文件（如 `.md` 或 `.txt`），深入理解项目的核心玩法、基调及限制条件。
2. **读取设计档案**：必须读取根目录下 `document/` 文件夹中的设计文档和规划文件，确保你的开发方案与现有的系统设计一致。
3. **缺失自动生成机制**：如果你在检索时发现根目录没有 `Game_Rules` 文件，或者没有 `document/` 文件夹及其内部的规划文档，你**必须主动新建这些目录和文件**，并根据当前上下文生成符合逻辑的基础模板，向我确认后再继续后续任务。

## 三、 进度同步工作流 (Progress Synchronization Workflow)
本项目采用闭环开发模式，你必须严格执行以下工作流：
1. **执行任务**：根据我的指令进行代码编写、Debug 或架构设计。
2. **强制状态更新**：在**每一次**完成一个具体功能、模块或修复后，你都必须立刻访问 `document/` 文件夹中的项目规划文档（例如 `Project_Schedule.md` 或 `Roadmap.md`）。
3. **同步记录**：在规划文档中勾选/标记已完成的事项，记录遇到的核心问题及解决方案，并更新下一步的开发计划。严禁在未同步文档的情况下直接结束对话。
4. **任务完成后进行修改汇报**：在完成一个具体功能、模块或修复后，你必须立刻向我汇报已完成的内容，包括代码修改、文档更新、测试结果等，如果涉及到新类的开发，必须告知我新类的名称、属性、方法等。

## 四、 数据驱动与配置建表规范 (Data-Driven & Table Configuration)
当开发中遇到以下类型的数据时，必须优先考虑建表（xlsx → JSON），而非硬编码在脚本中：
1. **大量配置数据**：如词缀库、技能库、装备底材、怪物模板等条目数量可能持续增长的数据。
2. **需要人为调控的数值**：如掉落权重、成长曲线、阈值参数、商店刷新概率等策划需要频繁调试的数值。
3. **关卡/内容设计配置**：如关卡波次、敌人组合、奖励池、剧情触发条件等后续关卡设计需要配置的数据。
4. **建表提示义务**：当你识别到上述场景时，**必须主动提醒我**："该数据适合建表，建议新建 xlsx 配置表"，并说明建议的表名、字段结构和类型，等待我确认后再继续。

## 五、 决策拦截与反幻觉机制 (Decision Interception & Anti-Hallucination)
在执行任何代码编写、架构重构或文档更新时，严禁基于猜测进行开发。你必须遵守以下“熔断”机制：
1. **禁止擅自假设**：如果我的需求描述存在歧义、缺失前置条件（如未指定具体的 Godot 节点类型、未明确性能预算等），或者涉及项目整体架构和玩法方向的重大选择，你**必须立即停止代码生成**。
2. **强制询问**：在停止生成后，你必须向我清晰地指出当前存在的歧义或缺失的信息。
3. **提供专业选项**：在提问时，不要仅仅抛出问题。你必须基于 Godot 引擎的特性或当前的 `Game_Rules`，提供 2-3 个可行的技术方向或解决方案供我选择，并客观列出每个方案的优缺点（如性能开销、开发成本、可维护性等），等待我的最终决策后再继续执行。

## 六、 Godot 专属文件与场景规范 (Godot-Specific File & Scene Protocols)
由于 Godot 序列化文件的敏感性，你在处理节点和场景时必须遵守以下红线：
1. **禁止直接修改场景/资源文件**：除非我明确授权，否则你**绝对不可**尝试生成或直接修改 `.tscn`、`.tres` 或 `project.godot` 文件。
2. **场景搭建指导**：当需求涉及添加新节点、设置节点层级或修改 Inspector 面板中的属性时，你必须提供明确的**编辑器 UI 操作步骤**（例如：“在 Scene 面板添加一个 RigidBody2D，并将其子节点 CollisionShape2D 的 Shape 设为 Circle”），而不是尝试通过代码生成这些结构。
3. **脚本与节点绑定**：你只负责输出 `.cs` C# 脚本代码。在代码顶部，必须通过注释明确标出该脚本应当挂载到什么类型的节点上（例如：`// 挂载于: CharacterBody2D`）。

## 七、 调试与排错工作流 (Debugging & Troubleshooting Standards)
在处理 Bug 和报错时，严禁基于猜测进行“盲改”：
1. **停止无意义的道歉与猜测**：如果代码运行报错，不要反复输出“抱歉，我修改一下”。
2. **索要堆栈信息**：你必须直接要求我提供 Godot Debugger 报错的完整堆栈信息（Stack Trace）以及相关的变量运行时状态。
3. **根因分析**：在给出修复代码前，必须先用一句话指出报错的根本原因（如“节点生命周期错误”、“空引用”或“信号连接时序问题”）。

## 八、 破坏性修改与版本控制确认 (Destructive Changes & VCS Check)
1. **重构预警**：当你的方案涉及跨文件修改、重构核心类、或是对 `Game_Rules` 基础机制进行大范围调整时，你必须在开始输出代码前给出高亮警告，并提醒我：“请确保你已在 Git 或其他版本控制工具中 Commit 当前进度”。
2. **原子化输出**：一次回答只专注解决一个模块，禁止一次性输出几百行混杂多个不同功能模块的代码。

## 九、 2D 物理、渲染与性能约束 (2D Physics, Rendering & Performance Limits)
作为 Godot 2D 项目的协作者，你在提供架构方案和脚本时，必须严格遵守以下底层逻辑红线：

1. **严格的物理生命周期**：所有涉及物理引擎的操作（如 `CharacterBody2D` 的移动、`RigidBody2D` 的受力、射线检测 `RayCast2D` 的强制更新），**必须且只能**在 `_PhysicsProcess(double delta)` 中执行。严禁将物理逻辑写在 `_Process(double delta)` 中导致帧率依赖和卡顿。
2. **精确的 2D 节点选型**：不要滥用物理节点。
   - 玩家/复杂逻辑敌人必须使用 `CharacterBody2D`，并通过 `MoveAndSlide()` 处理位移。
   - 纯物理驱动物件使用 `RigidBody2D`。
   - 仅用于检测重叠/触发器的区域，必须使用 `Area2D`。
3. **事件驱动与反轮询 (Anti-Polling)**：绝对禁止在 `_Process` 中高频轮询节点状态（例如每帧检查玩家血量是否为 0）。必须使用 Godot 的**信号 (Signals)** 机制进行事件驱动（C# 中通过 `[Signal]` 特性声明，`Connect()` 连接）。禁止在循环或 `_Process` 中调用高开销的 `GetNode()` 或 `FindChild()`，必须在 `_Ready()` 中缓存节点引用，或使用 `[Export]` + `[OnReady]` 关键字。
4. **UI 与世界坐标严格隔离**：在处理 HUD 和 UI（如血条、背包、暂停菜单）时，必须将其独立置于 `CanvasLayer` 节点之下，并严格使用 `Control` 类节点（如 `VBoxContainer`, `Label`）搭配锚点（Anchors）布局。绝不可将 UI 逻辑直接绑定到 `Node2D` 游戏世界坐标系中进行手动位移计算。
5. **内存管理与实例清理**：对于高频生成的 2D 实体（如子弹、粒子特效、飘字），必须确保其离开屏幕时（可配合 `VisibleOnScreenNotifier2D`）或生命周期结束时调用 `QueueFree()` 安全销毁。如果是弹幕类高密集生成需求，你必须主动提供**对象池 (Object Pool)** 的设计方案。
